# TM Stats Component Migration Matrix

## 1. Executive summary

- **Audit date:** 2026-07-16
- **Branch:** `redesign/tm-stats-dashboard-rebuild`
- **Starting revision:** `a9be6a0c4`
- **Inventory result:** 59 meaningful React components: 16 route/layout components under `src/app`, 6 shared components under `src/components`, and 37 feature components under `src/features`.
- **Shared ownership:** `AppShell` reaches all nine protected pages, `GroupSwitcher` reaches eight, and the chart/global-summary stack is reused across `/group`, `/insights`, and `/profile` through parent composites.
- **Overlap result:** 13 duplicate or overlapping responsibility clusters were found. The largest are score profile/scoring DNA, leaderboard/placement, style analytics, head-to-head, lineup effects, global analytics, and optional-data coverage.
- **High-risk result:** `InsightsDashboard` is the largest component at 1,016 lines. The next most migration-sensitive component families are log-game orchestration, corporation analytics, score-profile presentation, group/profile composites, replay, and final-action analysis.
- **Production changes:** none. This file records migration recommendations only; no route, component, query, formula, style, asset, or schema was changed.
- **Data-capability boundary:** repositories and types were inspected only to trace current component dependencies. Availability and feasibility classifications remain deferred to Step 0.3.

The inventory inspected every non-test `.tsx` file under `src/app`, `src/components`, and `src/features`; all 54 test/spec files were scanned and directly related tests are cited below. It also inspected the four CSS files, local public assets, and the related repositories, server actions, hooks/helpers, shared types, and asset paths in `analytics-repo`, `reference-repo`, the game/import/group repositories, the auth and Supabase clients, validation schemas, `build-insight-cards`, `finalize-game`, `use-log-game-draft`, OCR helpers, and theme tokens.

`StepHeading` and `SelectChevron` are intentionally excluded from the 59-component count because they are tiny, wizard-only presentational helpers. Route handlers and non-rendering calculation/repository helpers are also not counted as components, but architecturally significant helpers are called out in the risk and duplication sections.

### Matrix conventions

- **Runtime:** `S` = server component or server-compatible component; `C` = client component; `M` = mixed tree or server component with client children/server actions.
- **Scope:** uses the requested global, player, group, comparison, game, or shared classification. Authentication is classified as shared.
- **Tests:** “parent coverage” means the component has no direct test but is rendered by the cited parent test.
- **Styles:** “global/Tailwind” means `src/app/globals.css` tokens/classes plus inline Tailwind utilities.

Every inventory record includes category, current owner, scope, runtime, important props, data dependencies, tests, styling/assets, current responsibility, reusability, proposed destination, action, phase, risk, and known coupling.

## 10. Leaderboard

There is no dedicated leaderboard route or component. Ranking and placement are currently distributed across components already inventoried above:

| Current owner | Current leaderboard or placement responsibility | Future boundary |
| --- | --- | --- |
| `GroupDashboard` | Renders the first six weighted leaderboard rows, medal assets, win rate, and average placement. | Move formal rankings to `/leaderboard`; remove global/group-detail responsibilities from the ranking component. |
| `InsightsDashboard` | Renders a weighted-score comparison bar chart and player focus highlighting. | Move comparison behavior to `/compare`; do not make it a second leaderboard. |
| `ProfileDashboard` | Renders one player's weighted score, win rate, average placement, and score. | Retain a compact personal summary; deep placement analysis belongs in Individual Insights or Leaderboard. |
| `buildInsightCards` | Generates “Group Leader”/“Focused Form” prose from leaderboard rows. | Improvement/insight presentation should consume a centralized ranking definition rather than independently selecting the first row. |
| `GroupPage` and `AppShell` | `/group` is both the current page and the Leaderboard navigation target. | Phase 3 creates the route boundary; Phase 7 owns the dedicated ranking page and methodology. |

The migration must separate **ranking** (ordered methodology and eligibility) from **placement analysis** (average finish, distribution, matchup context). The current components often display both from the same `LeaderboardRow`, which makes them look like one responsibility even though the target architecture treats them differently.
